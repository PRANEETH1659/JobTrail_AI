# The Mentorship Contract

This document ensures that we (the IDE Assistant and the User) stick to a learning-first workflow for the JobTrail AI project. The goal is to ensure the user can confidently defend this project in an interview.

## The Problem

"I ask for code, receive code, paste it, run it, and move to the next step. I am not learning."

## The New Workflow: "Explain-First, Build-Second"

To break the copy-paste cycle, we will follow these strict rules for Phase 1 and Phase 2:

1. **No Data Dumps:** I will never generate massive, multi-file code blocks for you to paste.
2. **Conceptual Alignment:** Before writing code for a feature (e.g., User Login), I will explain the _concept_ and the _data flow_. I will ask you to confirm you understand the architecture before we write the code.
3. **Small Chunks:** We will write code iteratively, function by function.
4. **The Interview Check:** After every major component is built, I will pause development and act as an interviewer. I will ask you 1-2 technical questions about what we just built. We do not move forward until you can answer them.
5. **Debugging Together:** If there is an error, do not just ask for the fix. Paste the error, and I will explain _why_ the error is happening and ask _you_ how you think we should fix it.
6. **Daily Notion Summary:** At the end of every programming session, I will provide a formatted summary of our work. This summary will include what we built, challenges faced, why we chose specific technical approaches, and our overall progress. This will be ready to copy-paste into your Notion notebook for easy revision.

## The Tech Stack Rationale (Interview Prep)

- **React:** Component-based UI, makes state management easy.
- **Node/Express:** JavaScript everywhere. Easy to spin up REST APIs quickly.
- **MongoDB:** Document database. Job Descriptions and AI-generated outputs are highly unstructured/flexible text. A NoSQL JSON-like structure fits this better than a rigid SQL table.
- **JWT:** Stateless authentication. The server doesn't need to look up a session ID in the database for every single request, making it highly scalable.
